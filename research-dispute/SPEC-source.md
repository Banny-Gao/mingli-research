# 典籍原文录入规范

> **用途：** 规范典籍原文录入的格式与原则
> **原则：** 原文是整个数据管道的锚点，只录入，不解读，不分析
> **上游 SPEC：** SPEC-catalog.md（目录结构由 catalog 定义）
> **下游 SPEC：** SPEC-interpretation.md → SPEC-skill.md
> **适用范围：** 所有典籍

---

## 一、文件命名规范

```
books/{book-slug}/articles/{篇名}/source.md
```

- 篇名：使用中文全称，与目录一致
- 不得重名、空格、特殊字符

---

## 二、录入原则【核心】

### 2.1 原文照录

**只录入原文，不改变任何一个字。**

- 即使原文有明显错字，照录
- 不加译，不解释
- 不分段处理长段原文
- 异体字、避讳字、缺笔字照录原字形，必要时以 `【录入注】` 标记说明

### 2.2 版本意识

不同典籍、不同版本可能有差异，录入时：

- 明确标注来源版本（如有多个版本，选定一个后保持一致）
- 版本差异可在 `catalog.md` 元信息中说明

### 2.3 排版规范

- **段与段之间用空行分隔**
- 不添加额外的格式标记（无列表、无粗体、无斜体）
- 不添加额外的图片、表格

### 2.4 source 与 interpretation 的边界

**source.md 的职责：照录经典原文和历代注家文字，不做任何现代解读。**

| 归属 | 内容 | 原则 |
|------|------|------|
| source.md | 经典原文 + 历代注家注释（照录） | 一字不改，不加现代解读 |
| interpretation.md | 现代义理解读、分析、展开 | 基于 source 中的原文和注家文字 |

## X、字形策略

> **声明位置：** `books/{slug}/catalog.md` 元信息 blockquote 中
> **取值：** `原文照录`（默认） | `简体规范化`
> **作用域：** 该书 `source.md` 内容、目录名、`# 标题` 行、`catalog.md` 表格篇名

### X.1 两种策略

| 策略 | 含义 | 与 §2.1 关系 |
|------|------|-------------|
| `原文照录`（默认） | 1:1 对应印本，严字不改 | §2.1 全效生效 |
| `简体规范化` | 允许繁→简机械转换 + 歧义字语义复核 | §2.1 的"不改变任何一个字"对该策略做例外放行 |

### X.2 `原文照录` 工作流（默认）

§2.1 完全生效。异体字、避讳字、缺笔字、繁简差异字 一律照录原字形。适合版本有定本、校勘价值高的典籍。

### X.3 `简体规范化` 工作流

1. **机械转换**：使用标准 t2s 映射表（推荐 OpenCC `t2s`）对 source.md 逐字转换
2. **歧义字清单与复核**：LLM 列出歧义字映射清单初稿，人工 spot-check 关键出现处（按文意判定）：
   - `復` → `复`（重复）/ `覆`（覆盖）/ `複`（复杂）— 依语境
   - `像` → `像`（图像）/ `象`（象征/象棋）
   - `干` → `干`（干涉）/ `幹`（干部）/ `乾`（乾坤）
   - `后` → `后`（皇后/后面）/ `後`（之後）
   - `台` → `台` / `臺`（臺灣）
   - `余` → `余`（剩余）/ `餘`
   - `里` → `里`（公里）/ `裏`（裏面）
   - `几` → `几`（几个）/ `幾`（幾乎）
3. **目录与标题同步**：目录名、`# 篇名` 走同一转换
4. **catalog.md 同步**：表格篇名更新，元信息加 `字形策略：简体规范化`
5. **可追溯**：整本书转换合并为单次 commit（commit message 注明 `feat({书名}): 繁转简规范化`），便于审查与回滚

### X.4 标注方式

在 `books/{slug}/catalog.md` 的元信息 blockquote 中以一行声明：

````markdown
> 字形策略：简体规范化
````

未声明则视为 `原文照录`。

### X.5 与下游规范的关系

- `SPEC-interpretation.md`：解读时如需引用原文，从 source.md 现状引用，**不再回溯**到原印本繁简
- `generate.js`：路径推导基于 `articles/{篇名}/`，目录名变化由其自适应，无需改动
- `search-index.json`：重新生成即可，自动反映字形变化

## 三、结构规范

### 3.1 基本结构

每篇 source.md 应包含：

```
# 篇名

正文原文...
```

### 3.2 排版要求

- 一级标题仅篇名，无序号、无副标题
- 正文原文顶格书写，不加任何标记
- 注家注释以 `> 【注家名】` 开头，使用 markdown 块引用包裹
- 注释之间用空行分隔

---

## 四、勘误处理

若原文存在明显录入错误（而非原文本身的问题），不得直接修改：

```
> 【原注】原文文字...【录入注：原书此字疑误，待核】
```

---

## 五、禁区【红线】

1. **严禁在 source 文件中混入任何解读内容**
2. **严禁修改原文用字**
3. **严禁添加现代标点以外的任何标记** — 注家标记 `【注家名】`、`【录入注】` 和块引用 `>` 是编辑层约定的唯一例外，仅用于包裹注家标识和分层注文，不得在正文中新增其他标记
4. **严禁分段处理长段原文**
5. **严禁添加空行以外的任何内容**

---

## 六、与 generate.js 的关系

generate.js 按以下字段解析 source 文件：

```javascript
{
  chapter: string,      // 来自文件名（不含.md）
  title: string,       // 来自 # 篇名
  text: string,        // 全部原文内容
}
```

解析规则：
- 第一个 `#` 后的文字提取为 `title`
- `#` 之后到文件末尾为 `text`
