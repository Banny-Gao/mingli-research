# AI 可执行技能规范

> **用途：** 规范典籍篇章 `skill.md` 的产出格式
> **生成依赖：** `SPEC-source.md` + `SPEC-interpretation.md`（生成期间）
> **产出特性：** 生成后文件独立、自包含，可复制到其他地方使用
> **面向受众：** AI 执行者（Hermes Agent / Claude Code / 其他 LLM Agent）

---

## 一、前置判断

生成 skill 之前，先判断该篇章是否需要提炼技能。

**需要提炼的场景：**
- 有明确的判断逻辑（如命理的格局判定、医学的证型辨证、相术的面部特征归类等）
- 有可操作的流程（如命理的衰旺判断、医学的辨证施治步骤、卜筮的起卦断卦流程等）
- 有结构化输出（报告、列表、对比结果等）

**不需要提炼的场景：**
- 纯理论阐述，无操作流程
- 概念解释为主
- 内容过于抽象，难以结构化

只有判断需要时才生成 skill.md。

---

## 二、文件位置

```
books/{book-slug}/articles/{篇名}/skill.md
```

- 篇名：中文篇名，与 catalog.md 中的篇名严格一致
- 每篇只有一个 skill.md 文件
- **产出后独立使用**：文件内所有内容必须自包含，不依赖外部引用

---

## 三、文件结构

### 3.1 YAML Frontmatter

```yaml
---
displayName: 八格判定与病药分析
type: analysis
input: 八字四柱
output: 格局判定报告
description: 基于八格篇理论，对八字命局进行完整的格局判定与病药分析
---
```

> 以上为命理类示例。displayName 和 description 因书而异，type 和 input/output 格式保持一致。

**必填字段：**

| 字段 | 说明 | 示例 |
|------|------|------|
| `displayName` | 中文技能名称（用于展示和关联） | `三元分析法` |
| `type` | 技能类型 | `analysis` |
| `input` | 技能输入描述 | `八字四柱` |
| `output` | 技能输出描述 | `三元分析报告` |
| `description` | 一句话说明技能功能 | `基于天道篇的三元理论...` |

**type 可选值：**
- `analysis`：分析类（如命理格局判定、医学证型辨证、相术面相分析）
- `lookup`：查询类（如五行旺度查询、穴位定位、卦象查阅）
- `comparison`：对比类（如强弱对比、体质对比、方剂对比）
- `generation`：生成类（如行运推演、药方生成、卦辞推演）

### 3.2 功能定位

```markdown
## 功能

用一段话说明这个技能是做什么的、AI 执行者应该在何时调用它。不重复 interpretation 中的知识内容。

须注明目标读者：AI 执行 skill 后的输出是给**普通用户**看的，非专业研究者。术语和结论需附带通俗解释。
```

### 3.3 输入

```markdown
## 输入

列出所有需要的输入参数，每个参数含：名称、类型、格式约束、是否必填。

- 参数名：描述 | 类型 | 格式/约束 | 必填
```

**要求：**
- 每个输入参数必须有明确的类型（`string` / `number` / `string[]` / `object`）
- 有格式约束时必须注明（如"天干地支组合，2 字符"、"取值范围 1-12"）
- 可选参数标注默认值
- 输入不合法时在输出中体现（而非静默失败）

### 3.4 处理逻辑

```markdown
## 处理逻辑

按顺序列出分析/处理步骤。每个步骤标明：

1. **做什么** — 当前步骤的操作
2. **依据** — 引用原文注家中的关键依据。引用格式遵循 `SPEC-interpretation.md` §2.1 的注家标识规则和引文模板（`> 【注家标识】完整原文`），不使用 source.md 的原始格式
3. **判定** — 条件分支。

分支使用以下格式：

- 如果 **条件A**，则 **结果A**
- 否则如果 **条件B**，则 **结果B**
- 否则 **结果C**（兜底分支，必须有）
```

**要求：**
- 每一步必须有明确的输入→输出变换
- 所有条件分支**必须覆盖兜底分支（else/否则）**，禁止开放式分支
- 涉及多个因素综合判断时，须给出优先级权重
- 边界情况须显式处理（输入缺失、输入越界、无匹配规则时如何输出）

### 3.5 输出

```markdown
## 输出

定义输出格式。必须包含：

1. **输出模板** — 结构化格式（JSON Schema / Markdown 模板 / 字段列表），标明每个字段的类型和含义
2. **字段说明** — 逐一解释每个输出字段
```

**要求：**
- 输出模板定义 AI 应产出的**内容结构**，AI 执行时按模板填充
- 每个输出字段有明确的类型和取值范围
- **输出面向普通用户** — 每个技术术语须附带一句通俗解释，结论须说明"这意味着什么"
- 对于分析/判定类 skill，输出中必须包含**判定依据**字段（引用哪个原文/注家），不可只给结论
- 输出末尾建议附"进一步了解"指引（可阅读的相关篇章）
- **自包含性**：所有通俗解释必须在 skill 文件内部产出，不依赖外部引用

### 3.6 示例验证

```markdown
## 示例

至少提供一个完整的输入→输出示例，用于验证技能正确性。

### 示例输入
- 参数1：值
- 参数2：值

### 示例输出
（完整的输出内容，与 §3.5 模板一致）

### 验证要点
- 步骤 X 应触发 Y 分支，因输入满足条件 Z
```

**要求：**
- 至少 1 个示例，推荐 2-3 个覆盖主路径和边界
- 示例必须可复现：给定示例输入，按处理逻辑执行应得到示例输出
- 示例应标注验证要点

---

## 四、设计原则

1. **基于原文+解读** — skill 从 source（原文依据）和 interpretation（义理解读）中提炼可执行能力
2. **生成后独立** — 产出文件必须自包含，可复制到其他地方使用，不依赖外部引用
3. **功能导向** — 每个 skill 是一个 AI 可独立调用的能力单元
4. **引用原文但不堆砌** — 引用关键原文作为依据
5. **输出可结构化** — 输出应能被程序解析和消费
6. **不做 speculative 设计** — 只定义当前篇章明确支持的能力

## 五、产出自检清单

- [ ] displayName 已填写（中文，用于展示和关联）
- [ ] type 正确（analysis / lookup / comparison / generation）
- [ ] 功能定位清晰，说明何时调用
- [ ] 输入参数有类型和格式约束，必填/可选标注清楚
- [ ] 处理逻辑步骤完整，条件分支均有兜底 else，无边界的开放式判定
- [ ] 边界情况显式处理（输入缺失/越界/无匹配）
- [ ] 输出格式结构化，每个字段有类型和含义说明
- [ ] 判定类输出包含依据字段（引用原文），非仅结论
- [ ] **输出内容自包含**：通俗解释全部在文件内部产出，不依赖外部
- [ ] 至少 1 个示例输入→输出，示例可复现且有验证要点
- [ ] 基于 source + interpretation 提炼，而非简单复制
- [ ] 引用原文注家有来源标注（使用块引用格式）
