# format-check Skill 设计文档

**日期**：2026-07-02
**状态**：待审批
**触发**：基于 5 篇 articles 随机抽样分析，总结 markdown 排版格式问题

## 1. 背景与动机

项目内 `books/` 下共 2545 篇 markdown 文章，通过 react-markdown 渲染（配置 `remarkGfm` + `rehypeRaw` + `rehypeSlug` + `rehypeHighlight` + `rehypeAutolinkHeadings`）。随机抽检 5 篇发现：标题扁平、引用块被代码块包裹导致渲染错误、古典韵文换行丢失、同类条目无子标题等问题普遍存在。

需要创建一个 `format-check` skill 来系统性地检测和修复这些问题。

## 2. 核心定位

一个 markdown 格式规范检查与修复 skill，针对 `books/` 下所有文章，确保在项目 react-markdown 配置下渲染效果良好。

## 3. 模式与调用

### 3.1 两种执行模式

| 模式 | 命令 | 行为 |
|------|------|------|
| **交互式修复** | `/format-check` | 逐篇扫描 → 逐问题展示 → 用户确认 → 应用修复 |
| **自动修复** | `/format-check --fix` | 扫描 → 直接修复（非破坏性规则），含 LLM 的规则跳过 |

### 3.2 两种范围

| 范围 | 命令 | 说明 |
|------|------|------|
| **单篇** | `/format-check <file-path>` | 指定一篇 md 文件路径 |
| **按书** | `/format-check --book <slug>` | 指定一本书，如 `--book 滴天髓阐微` |

### 3.3 LLM 分析模式

| 触发方式 | 行为 |
|----------|------|
| `/format-check --analyze` | 对指定范围启用 LLM 深度分析（智能分段 + 标题建议） |
| 交互模式下检测到 R4 | 询问用户是否启用 LLM 分析 |
| `--fix` 模式 | 不触发 LLM |

## 4. 检查规则（15 条）

### 🔴 严重（自动修复 / --fix 可修复）

| # | 规则 | 说明 |
|---|------|------|
| R1 | 韵文/口诀换行丢失 | 古典诗词、歌诀用单换行分隔，CommonMark 下单换行是 soft break 渲染为空格。修复：转双换行（段落）或行尾两空格（硬换行） |
| R2 | 代码块内包含引用标记 `>` | ` ``` ` 包裹含 `>` 的内容渲染为 `<pre><code>`，但内容意图是原文引用。`--fix` 跳过（不自动修改），交互模式询问是否转为 `>` 引用块 |
| R14 | Mermaid 语法错误 | 未定义节点的引用、不闭合的括号、孤立节点等，导致渲染失败。`--fix` 尝试修复已知模式（补全缺失的节点定义、修复 typo） |

### 🟡 警告（交互确认 / --fix 自动修复）

| # | 规则 | 说明 |
|---|------|------|
| R3 | 标题层级过浅 | 仅有 `#` 无 `##`/`###`，长文无视觉锚点和锚点链接。修复：LLM 分析内容后生成子标题 |
| R4 | 同类条目无子标题 | 多个同类条目（如神煞、命造）仅靠 `【】` 标记区分，无 `###` 分区。修复：LLM 分析后为每个条目生成子标题 |
| R5 | 标题跳级 | `#` 后直接出现 `###` 跳过 `##`，破坏大纲树结构。修复：降级为 `##` |
| R6 | 长引用块无内部结构 | 单段引用超过 15 行，缺乏分段/加粗/列表。修复：插入空行分段，关键术语加粗 |
| R7 | 引用块内部多段内容被挤成单段 | 引用块内多段用单换行分隔，渲染后合并。修复：插入空行分隔 |
| R15 | 流程图可读性 | 综合检查 Mermaid 图的可读性，含移动端适配（子项见下表）。LLM 分析后给出优化建议，用户确认后应用 |

**R15 子检查项：**

| 子项 | 检测条件 | 修复建议 |
|------|----------|----------|
| 节点过多 | 单图节点 > 15 个 | 拆分为多张图或用 subgraph 分组 |
| 方向不当 | `graph LR` + 节点 > 8 个 | 建议改为 `graph TD`，LR 在移动端横向溢出 |
| 缺少 subgraph | 有天然分组的节点未用 subgraph | 用 subgraph 包裹，提升层次感 |
| 节点标签过长 | 单个节点文字 > 20 字 | 缩短标签，详细说明放图外 |
| 缺少图标题 | 图上方无 `**图X：xxx**` 描述 | 添加描述性标题 |
| **移动端横向溢出** | `graph LR` 方向 + 层级 > 4 层 | 🔴 严重，移动端必溢出，强制建议改 TD |
| **移动端节点密度** | 总节点 > 10 且方向为 LR | 🟡 警告，移动端阅读困难 |

### 🔵 建议（仅交互提示，不自动修复）

| # | 规则 | 说明 |
|---|------|------|
| R8 | 可用表格未用 | 对比性内容（如命造对照）用纯文本排列，可转为 GFM 表格 |
| R9 | 缺少分节线 | 大章节之间无 `---` 视觉分隔 |
| R10 | 连续空行冗余 | 3+ 连续空行，规范为 1-2 行 |
| R11 | 中英文间缺空格 | 中文与英文/数字混排缺少空格，影响阅读体验 |
| R12 | 强调标记不一致 | 关键术语有的用 `**粗体**` 有的无标记。建议首次出现时统一标记 |
| R13 | 行尾空白 | 行尾多余空格，git diff 噪音 |

## 5. LLM 分析层

独立于规则表，用于需要内容理解才能执行的修复：

- **触发**：`--analyze` 标志，或交互模式下用户确认
- **行为**：全文理解 → 智能分段 → 生成 `###` 标题建议 → 用户确认 → 写入
- **适用规则**：R3（标题层级过浅）、R4（同类条目无子标题）、R15（流程图可读性）
- **不适用**：`--fix` 自动模式（避免不可控修改）

## 6. 文件结构

```
.claude/skills/format-check/
├── SKILL.md              # 主入口，路由 + 引导
├── rules/
│   ├── critical.md       # R1-R2 + R14 严重规则
│   ├── warning.md        # R3-R7 + R15 警告规则
│   └── suggestion.md     # R8-R13 建议规则
└── shared/
    ├── entrypoint.md     # 引导式状态机
    ├── scanner.md        # 规则扫描引擎
    └── fixer.md          # 修复执行器
```

## 7. 执行流程

```
用户输入 /format-check [--fix] [--analyze] [--book <slug>|<file>]
    │
    ├─ Step 1: 确定范围和模式
    │     ├─ 单篇：读取指定文件
    │     └─ 按书：find books/{slug}/articles/**/*.md
    │
    ├─ Step 2: 逐文件扫描
    │     ├─ 应用 R1-R15 规则
    │     └─ 收集问题列表，按严重度分组
    │
    ├─ Step 3: 模式分发
    │     ├─ --fix：自动修复 R1 + R5-R7 + R10 + R13 + R14（跳过 R2-R4 + R8-R9 + R11-R12 + R15 + LLM）
    │     └─ 交互：逐问题展示，用户逐条确认
    │
    ├─ Step 4: LLM 分析（可选）
    │     └─ 仅 --analyze 或交互下用户确认后触发
    │
    └─ Step 5: 输出报告
          └─ 汇总：文件数、问题数、修复数、跳过数
```

## 8. 非目标

- 不做语法校验（那是 lint 工具的事）
- 不做内容质量审查（如语义错误、逻辑矛盾）
- 不修改 `interpretation.md` 的解读内容本身
- 不修改 `source.md` 的原文内容本身（仅调整 markdown 标记层）

## 9. 与现有 skill 的关系

| 现有 skill | 关系 |
|-----------|------|
| `self-check` | 互补。self-check 检查内容合规（SPEC 规范），format-check 检查格式排版 |
| `interpretation-create` | 可集成。生成 interpretation.md 后自动触发格式检查 |
| `source-create` | 可集成。生成 source.md 后自动触发格式检查 |
