# AI 可执行技能规范

> **用途：** 规范典籍篇章 `skill.md` 的产出格式
> **版本：** 1.2 — 通用化：示例和前置判断覆盖山医命相卜
> **上游 SPEC：** SPEC-interpretation.md（需先完成 source + interpretation）
> **面向受众：** AI 执行者（Hermes Agent / Claude Code / 其他 LLM Agent）

---

## 前置判断

生成 skill 之前，先判断该篇章是否需要提炼技能：

**需要提炼的场景：**
- 有明确的判断逻辑（如命理的格局判定、医学的证型辨证、相术的面部特征归类等）
- 有可操作的流程（如命理的衰旺判断、医学的辨证施治步骤、卜筮的起卦断卦流程等）
- 有结构化输出（报告、列表、对比结果等）

**不需要提炼的场景：**
- 纯理论阐述，无操作流程
- 概念解释为主
- 内容过于抽象，难以结构化

只有判断需要时才生成 skill.md。

---

## 一、文件位置

```
books/{book-slug}/articles/{篇名}/skill.md
```

- 篇名：中文篇名，与 catalog.md 中的篇名严格一致
- 每篇只有一个 skill.md 文件

## 二、文件结构

### 2.1 YAML Frontmatter

```yaml
---
displayName: 八格判定与病药分析
type: analysis
input: 八字四柱
output: 格局判定报告
description: 基于八格篇理论，对八字命局进行完整的格局判定与病药分析
---
```

> 以上为命理类示例。displayName 和 description 因书而异，type 和 input/output 格式保持一致。

**必填字段：**

| 字段 | 说明 | 示例 |
|------|------|------|
| `displayName` | 中文技能名称（用于展示和关联） | `三元分析法` |
| `type` | 技能类型 | `analysis` |
| `input` | 技能输入描述 | `八字四柱` |
| `output` | 技能输出描述 | `三元分析报告` |
| `description` | 一句话说明技能功能 | `基于天道篇的三元理论...` |

**type 可选值：**
- `analysis`：分析类（如命理格局判定、医学证型辨证、相术面相分析）
- `lookup`：查询类（如五行旺度查询、穴位定位、卦象查阅）
- `comparison`：对比类（如强弱对比、体质对比、方剂对比）
- `generation`：生成类（如行运推演、药方生成、卦辞推演）

### 2.2 功能定位

```markdown
## 功能

用一段话说明这个技能是做什么的、AI 执行者应该在何时调用它。不重复 interpretation 中的知识内容。

须注明目标读者：AI 执行 skill 后的输出是给**普通用户**看的，非专业研究者。术语和结论需附带通俗解释。
```

### 2.3 输入

```markdown
## 输入

列出所有需要的输入参数，每个参数含：名称、类型、格式约束、是否必填。

- 参数名：描述 | 类型 | 格式/约束 | 必填
```

**要求：**
- 每个输入参数必须有明确的类型（`string` / `number` / `string[]` / `object`）
- 有格式约束时必须注明（如"天干地支组合，2 字符"、"取值范围 1-12"）
- 可选参数标注默认值
- 输入不合法时在输出中体现（而非静默失败）

### 2.4 处理逻辑

```markdown
## 处理逻辑

按顺序列出分析/处理步骤。每个步骤标明：

1. **做什么** — 当前步骤的操作
2. **依据** — 引用原文注家中的关键依据（标记格式以各书 source.md 为准，使用 `> 【注家名】原文` 块引用）
3. **判定** — 条件分支。

分支使用以下格式：

- 如果 **条件A**，则 **结果A**
- 否则如果 **条件B**，则 **结果B**
- 否则 **结果C**（兜底分支，必须有）
```

**要求：**
- 每一步必须有明确的输入→输出变换
- 所有条件分支**必须覆盖兜底分支（else/否则）**，禁止开放式分支
- 涉及多个因素综合判断时，须给出优先级权重（哪个因素优先于哪个）
- 边界情况须显式处理（输入缺失、输入越界、无匹配规则时如何输出）

### 2.5 输出

```markdown
## 输出

定义输出格式。必须包含：

1. **输出模板** — 结构化格式（JSON Schema / Markdown 模板 / 字段列表），标明每个字段的类型和含义
2. **字段说明** — 逐一解释每个输出字段
```

**要求：**
- 输出模板定义 AI 应产出的**内容结构**，AI 执行时按模板填充
- 每个输出字段有明确的类型和取值范围
- **输出面向普通用户** — 每个技术术语须附带一句通俗解释，结论须说明"这意味着什么"
- 对于分析/判定类 skill，输出中必须包含**判定依据**字段（引用哪个原文/注家），不可只给结论
- 输出末尾建议附"进一步了解"指引（可阅读的相关篇章）

### 2.6 示例验证

```markdown
## 示例

至少提供一个完整的输入→输出示例，用于验证技能正确性。

### 示例输入
- 参数1：值
- 参数2：值

### 示例输出
（完整的输出内容，与 §2.5 模板一致）

### 验证要点
- 步骤 X 应触发 Y 分支，因输入满足条件 Z
```

**要求：**
- 至少 1 个示例，推荐 2-3 个覆盖主路径和边界
- 示例必须可复现：给定示例输入，按处理逻辑执行应得到示例输出
- 示例应标注验证要点，说明为什么这个输入产生这个输出

## 三、设计原则

1. **基于原文+解读** — skill 从 source（原文依据）和 interpretation（义理解读）中提炼可执行能力，三者相互支撑
2. **功能导向** — 每个 skill 是一个 AI 可独立调用的能力单元
3. **引用原文但不堆砌** — 引用关键原文作为依据
4. **输出可结构化** — 输出应能被程序解析和消费
5. **不做 speculative 设计** — 只定义当前篇章明确支持的能力

## 四、完整示例

> 以下为命理类技能示例。其他术数（医/相/山/卜）参照相同结构，具体术语和依据因书而异。

```yaml
---
displayName: 八格判定与病药分析
type: analysis
input: 八字四柱
output: 格局判定报告
description: 基于八格篇理论，对八字命局进行完整的格局判定与病药分析
---

## 功能

根据月令取格规则，判断八字格局，识别用神喜忌，分析病药所在。调用时机：用户提供完整八字四柱后，需要判定格局和用神方向时调用。

## 输入

- 年柱：天干地支组合 | string | 2 字符，如"庚辰" | 必填
- 月柱：天干地支组合 | string | 2 字符，如"癸未" | 必填
- 日柱：天干地支组合 | string | 2 字符，如"乙未" | 必填
- 时柱：天干地支组合 | string | 2 字符，如"癸未" | 必填

## 处理逻辑

1. **月令定格** — 取月支，对照十二月令地支表确定月令主气
   - 依据：`> 【原注】月令者，提纲也。格从月起，月支所藏为格之根本。`
   - 判定：
     - 如果 **月支为寅**，则 主气为甲木
     - 如果 **月支为卯**，则 主气为乙木
     - ...（每个地支均需覆盖）
     - 否则 **输入月柱格式有误，返回错误提示**

2. **透干取格** — 检查月支藏干是否在天干透出
   - 依据：`> 【任氏曰】月支藏干透于天干者，取以为格。`
   - 判定：
     - 如果 **月支主气在年/月/日/时干中透出**，则 取该主气为格
     - 否则如果 **月支中气在天干透出**，则 取中气为格
     - 否则 **取月支本气为格**（兜底）

3. **用神判定** — 依据格局和日主强弱确定用神
   - 依据：`> 【任氏曰】格有成败，用神即治病之药。`
   - 判定：
     - 如果 **日主强 + 格局为印/比**，则 用神取财/官/食
     - 如果 **日主弱 + 格局为财/官/食**，则 用神取印/比
     - 否则 **以扶抑中和为原则，取平衡五行**

4. **病药分析** — 识别破格之病与成格之药，给出调候建议

## 输出

AI 执行时按以下模板产出内容，**面向普通用户**，术语须附带通俗解释。

```
## 格局判定报告

- **格局名称**：string — 一句话说明这个格局意味着什么
- **取格依据**：string — 引用原文，用通俗语言解释为什么取这个格
- **日主强弱**：string（"强" | "弱" | "中和"）— 强弱对命主的影响（通俗）
- **用神**：string — 用神是什么 + 为什么选它
- **喜神**：string — 喜神是什么 + 为什么喜它
- **忌神**：string — 忌神是什么 + 为什么忌它
- **病药分析**：string — 命局的"问题"和"解法"，用比喻说清楚
- **判定置信度**：string（"确定" | "参考"）
- **进一步了解**：string — 推荐阅读的相关篇章（1-2 篇）
```

## 示例

### 示例输入
- 年柱：庚辰
- 月柱：癸未
- 日柱：乙未
- 时柱：癸未

### 示例输出

## 格局判定报告

- **格局名称**：偏印格
  偏印格的人通常善于深度思考和学习，但也容易思虑过多。

- **取格依据**：月支为未，未的主气是己土、中气是丁火、余气是乙木。己土和丁火都没有在天干出现，但乙木透出在日干（命主自己），因此取偏印格。简单说：月份决定了格局的"底色"是未土，而日干乙木是真正发挥作用的力量。

- **日主强弱**：弱
  乙木坐在未土上失令（未月土旺木囚），年柱庚金正官克制日主，虽有水来生木但力量不足。日主弱意味着命主需要"帮扶"——就像一个人力量不够，需要有人帮忙。

- **用神**：水（印星）
  印星可以生扶日主，同时化解官杀的克制。选水是因为水既能生木（让命主变强），又能泄金（减少压力）。

- **喜神**：木（比劫）
  同类相助，木可以帮木分担压力。

- **忌神**：金（官杀）
  金会克制日主乙木，让本来就弱的命主更受压。大运流年见金时要多加注意。

- **病药分析**：这个命局的"病"是日主偏弱加上官杀（金）克制——就像一个身体底子偏弱的人还背着压力。"药"是用水（印星）来化解——水一方面能生木让命主强壮，另一方面能泄金减少压力来源。月干癸水就是偏印，本身是药，但癸水坐在未土上被克制，药力不够，需要大运走到水旺的年份才能真正发挥效果。

- **判定置信度**：确定

- **进一步了解**：推荐阅读《滴天髓阐微》「方局」「配合」篇，了解格局与大运的配合关系。

### 验证要点
- 步骤 1：月支未 → 主气己土，中气丁火，余气乙木 ✓
- 步骤 2：己土、丁火均未透干，乙木透于日干 → 触发"中气/余气透干"分支 ✓
- 步骤 3：日主乙木坐未失令 + 官杀克身 → 判定"弱"，触发"日主弱"分支 → 用神取印比 ✓
```

## 五、产出自检清单

- [ ] displayName 已填写（中文，用于展示和关联）
- [ ] type 正确（analysis / lookup / comparison / generation）
- [ ] 功能定位清晰，说明何时调用

- [ ] 处理逻辑步骤完整，条件分支均有兜底 else，无边界的开放式判定
- [ ] 边界情况显式处理（输入缺失/越界/无匹配）
- [ ] 输出格式结构化，每个字段有类型和含义说明
- [ ] 判定类输出包含依据字段（引用原文），非仅结论
- [ ] 至少 1 个示例输入→输出，示例可复现且有验证要点
- [ ] 基于 source + interpretation 提炼，而非简单复制
- [ ] 引用原文注家有来源标注（使用块引用格式）

## 六、与 catalog 的关联

catalog.md 中关联技能时，使用 `displayName`：

```markdown
| 01 | 天道 | 已解读 | 三元分析法 |
```

同一本书内 `displayName` 必须唯一，作为关联查找的 key。

`displayName` 必须与 catalog.md 中该篇章对应行的"关联技能"列完全一致。如果 catalog 中已填写关联技能，skill.md 的 displayName 必须匹配；如果 skill.md 产出后 displayName 有变更，需同步更新 catalog.md。
