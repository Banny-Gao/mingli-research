## 四、Subagent 执行模式

### 4.1 设计

每批由 1 个独立 subagent 执行，subagent 类型：`general-purpose`。prompt 自包含，SPEC 通过文件引用而非内联。

### 4.2 并行度

**6 并行**（typical max），分轮执行。本书总批数 = 48 批 = 8 轮 × 6。

### 4.3 Subagent Prompt 模板

```text
你是 子平法解读 subagent，负责为本批 N 篇古籍生成 interpretation.md。

【书目】渊海子平
【批号】{batch_id}
【篇目】（源文件路径 books/渊海子平/articles/{篇名}/source.md）
- {篇名1}
- {篇名2}
- ... 共 N 篇

【严格遵循规范】
- 完整解读规范：research-dispute/SPEC-interpretation.md
- 完整通用规范：research-dispute/SPEC-general.md
- 重点：文件禁"升华洞见"；自评分内部不写入；反模板禁止直套
- 子平特色：诗诀高度浓缩，术语必须当场释义（详见 §七.2）

【输出去向】
books/渊海子平/articles/{篇名}/interpretation.md（每篇 1 个文件）

【完成后报告】
- 已生成文件列表（绝对路径）
- LLM 合规自检评分（内部 0-5，不写入文件）
- 触红线项（如有）
- 任何卡点 / SPEC 改进建议
```

### 4.4 调度流程（每轮）

```
1. 调度方在同一 message 发出 6 个 subagent 调用
2. 每个 subagent 独立完成批内 5-6 篇
3. 全部完成后调度方汇总到 docs/解读日志.md
4. 更新本计划 §三 状态列
5. 下一轮
```

### 4.5 监控指标

- 每批跑批时间（5-10 min 正常）
- 自评合规分均值（≥ 4.5 视为合格）
- 触红线频次（持续高则需重新审视 SPEC）
- 跨批重复论点检测（避免批间冗余）

## 六、风险与变更

### 6.1 已知风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 批次多（48 批）跑完周期长 | 中途 SPEC 可能再迭代 | 每 10 批 checkpoint，重新校验 SPEC 一致性 |
| 大量短篇（诗诀）密集同批，LLM 易模板化 | 反模板规则触发频繁 | 短篇批次强制结构自决，不允许批量套诗诀模板 |
| 与滴天髓阐微章节命名重叠（如正官格、正官论） | 容易跨书混淆 | LLM 不得做跨书断言（SPEC §1.2 §2 ❌ #5 升级版） |
| 模式判据 `mode_of()` 与实际字符数偏差 | 章节层级与预判不符 | LLM 必须以运行时计算为准，不盲信预判 |
| 6 subagent 并行 token 成本 | 单轮消耗约 6× 单批 token | 按需降低并行度（如 3-4），或按轮分批执行 |
| 49→48 重排导致与早期进度记录不一致 | 跟踪脱节 | 旧记录的 49 批视为历史快照，新进度按 48 批重计 |

### 6.2 计划变更规则

- 篇目调整：在本文件 §三 表中编辑，commit 时附理由
- 批次数量调整：仅当 SPEC 重大修订或篇目整体重排时启用；调整后需重新校验 48 批总和 = 243
- 模式预判偏差：仅注释，不阻断
- 并行度调整：在 §四 记录
