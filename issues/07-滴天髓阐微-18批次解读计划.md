## 四、Subagent 执行模式

### 4.1 设计

每批由 1 个独立 subagent 执行，subagent 类型：`general-purpose`。prompt 自包含（~500 tokens），SPEC 通过文件引用而非内联。

### 4.2 并行度

**6 并行**（typical max），分轮执行。本书总批数 = 18 批 = 3 轮 × 6。

总任务：三本书共 84 批 = 14 轮（滴天 3 + 紫微 3 + 渊海 8 = 14 轮），可按书目顺序串跑或混合并行。

### 4.3 Subagent Prompt 模板

```text
你是 子平学解读 subagent，负责为本批 N 篇古籍生成 interpretation.md。

【书目】滴天髓阐微
【批号】{batch_id}
【篇目】（源文件路径 books/滴天髓阐微/articles/{篇名}/source.md）
- {篇名1}
- {篇名2}
- ... 共 N 篇

【严格遵循规范】
- 完整规范：research-dispute/SPEC-interpretation.md（2026-06-04 修订）
- 重点 4 条：① 开头 50 字结构说明；② 文件禁"升华洞见"；③ 自评分内部不写入；④ 反模板禁止直套

【输出去向】
books/滴天髓阐微/articles/{篇名}/interpretation.md（每篇 1 个文件）

【完成后报告】
- 已生成文件列表（绝对路径）
- LLM 自评合规分（内部 0-5，不写入文件）
- 触红线项（如有）
- 任何卡点 / SPEC 改进建议
```

### 4.4 调度流程（每轮）

```
1. 调度方在同一 message 发出 6 个 subagent 调用
2. 每个 subagent 独立完成批内 3-4 篇
3. 全部完成后调度方汇总到 docs/解读日志.md
4. 更新本计划 §三 状态列
5. 下一轮
```

### 4.5 监控指标

- 每批跑批时间（5-10 min 正常）
- 自评合规分均值（≥ 4 视为合格）
- 触红线频次（持续高则需重新审视 SPEC）
- 跨批重复论点检测（避免批间冗余）

## 六、风险与变更

### 6.1 已知风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 模式判据 `mode_of()` 与现有 60 篇实际字符数偏差 | 章节层级与预判不符 | LLM 必须以运行时计算为准，不盲信预判 |
| 批次内跨篇关联断言（SPEC §1.2 §2 ❌ #5） | 容易触发红线 | LLM 只能基于本篇原文证据做笼统定位 |
| 批次 1 旧产出已删，新产出需严格按新 SPEC | 重做时需特别注意新加的 4 条规则 | 重做前再读一遍 SPEC §1.2 §2 和 §七 |
| 6 subagent 并行的 token 成本 | 单轮消耗约 6× 单批 token | 按需调整并行度（如降到 3-4），或按轮分批执行 |

### 6.2 计划变更规则

- 篇目调整：在本文件 §三 表中编辑，commit 时附理由
- 批次数量调整：仅当 SPEC 重大修订或篇目整体重排时启用；调整后需重新校验 18 批总和 = 63
- 模式预判偏差：仅注释，不阻断
- 并行度调整：在 §四 记录，commit 时附理由
