# 【产品审视】知识库现状问题报告

> 日期：2026-05-18
> 视角：资深产品专家（不含运营/商业化维度）
> 状态：Open

---

## 一、产品定位

**《滴天髓》学术知识库**：面向命理学研究者的古籍原文+注疏解读+可执行技能的数字阅读平台。

> 关键词：研究者、古籍、数字阅读平台。

---

## 二、目标用户分层与真实需求

| 用户                 | 真实需求                           | 当前满足程度       |
| -------------------- | ---------------------------------- | ------------------ |
| 命理爱好者（初学者） | 想学滴天髓，但看不懂原文           | 基本满足           |
| 命理师（从业者）     | 查阅案例、验证理论                 | 部分满足           |
| **命理学研究者**     | **引用原文、追溯版本、做学术论证** | **几乎不满足**     |
| AI/开发者            | 调用技能做命理工具                 | 有框架但无实质内容 |

### 核心问题

研究者真正需要的是：

1. **原文溯源**：出处、版本异文、不同注家的分歧
2. **交叉索引**：篇与篇之间的理论关联
3. **快速定位**：一段话能否直接引用、精确位置在哪
4. **版本对比**：任铁樵注 vs 徐乐吾注 vs 梁湘润注的差异

**当前产品在这四点上几乎是空白。**

---

## 三、体验问题

### 3.1 信息架构硬伤

> **用户来这里是为了找一段话，不是因为想读某一篇。**

研究者的典型场景：脑子里有一个问题——"滴天髓里讲'旺极宜泄'的原文在哪里？"

当前路径：

```
Landing → 选择书籍 → 阅读目录 → 猜在哪篇 → 打开篇章 → 在整篇里找那句话
```

**缺失检索系统的数字阅读平台，对研究场景是致命缺陷。**

### 3.2 内容体验的设计问题

| 问题                               | 说明                                                                                                                                                |
| ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **解读是"教科书式"的**             | 当前 interpretation.md 结构（入门导读→原文精读→义理解析→命造拆解→总结）是教学设计，不是查阅设计。研究者在找"用神不可伤"的原文，不会去看"入门导读"。 |
| **原文和解读没有锚点跳转**         | 用户读原文时遇到不懂的句子，需要手动切换文件，没有行内锚点。                                                                                        |
| **技能系统是"展示型"非"可执行型"** | skill.md 存在于文件系统，但前端完全不可见。用户感知不到差异化价值。                                                                                 |
| **技能与解读边界模糊**             | 部分 skill.md 实际上是 interpretation.md 的精简版，内容高度重复。                                                                                   |

### 3.3 图解问题

当前图解规范要求"三列流程图""树状图"，但实际产出：

- 内容多为"核心概念罗列"而非"逻辑推导"
- 图解结论与正文完全一致，无增量信息
- 图解变成"美观要求"而非"思维呈现"

## 四、内容质量问题

| 问题                 | 说明                                                                                                                                        |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 解读两极分化         | 好的：原文引用规范、术语有加注、有命造案例。问题：3000-5000字长文信息密度高但可读性差；"常见误解"混入了"学术争议"概念；命造实例格式不统一。 |
| 原文来源约束是双刃剑 | "原文路径唯一来源：滴天髓阐微目录.html"是好的质量控制，但代价是：当源文件有错误时，整个知识库跟着错。没有校验层或版本异文记录。             |
| 版本异文缺失         | 无任何版本差异记录，与"学术级"定位严重不符。                                                                                                |

## 五、技术架构问题

### 5.1 generate.js 承担过多角色

当前设计：Markdown → 编译 → TypeScript → 前端

- 优点：静态生成，加载快
- 缺点：任何内容更新需重新编译；前端与内容之间多了一层编译层；不支持动态内容（用户注释、版本对比）

### 5.2 数据模型扩展性不足

```typescript
// 当前 ChapterInfo
num, name, isDone, category

// 研究级知识库需要
num, name, isDone, category, difficulty, relatedArticles[], tags[], sourceVersion, lastUpdated
```

当前模型是"展示驱动"而非"数据驱动"。

### 5.3 搜索能力缺失

> 这可能是整个产品最大的功能空白。

一个没有搜索功能的古籍知识库，就像一个没有索引的图书馆。

## 七、战略性问题

### 7.1 "学术级"是标签不是能力

产品自称"学术级古籍阅读平台"，但：

- 没有引用格式支持（学术平台基本功能）
- 没有版本记录（学术平台核心功能）
- 没有学术社区（研究者需要同行交流）

### 7.2 技能系统战略错位

> skill.md 是给 AI 执行者用的，不是给人类用户用的。

这意味着产品同时服务两个完全不同的用户，但路径完全分离。AI 技能的价值对人类用户完全不可见。

### 7.3 内容深度 vs 覆盖广度

只做滴天髓，做到极致，也只是一个垂直领域的工具。护城河不在于深度，而在于网络效应——链接越多、引用越多、平台越权威。

## 八、优先级建议

### P0（必须做，现在就有）

- [ ] **全文搜索功能** — 篇内定位、关键词高亮。没有这个，研究场景无法成立。
- [ ] **原文-解读锚点跳转** — 读原文时一键跳到对应解读段落，反向亦然。

### P1（重要，三个月内做）

- [ ] **篇目关联图** — 每篇的"关联篇章"从文字列表升级为可视化关系图。
- [ ] **段内引用标注** — 每个段落有稳定引用 ID，支持学术直接引用。
- [ ] **版本异文记录** — 在 source.md 中记录不同版本的文字差异。

### P2（差异化，一年内做）

- [ ] **AI 辅助解读入口** — 用户针对某段落提问，AI 基于 interpretation + skill 生成回答。
- [ ] **学者笔记层** — 用户在篇章上做个人笔记，形成个人知识库。
- [ ] **内容扩展** — 从滴天髓扩展到渊海子平、三命通会等，建立命理古籍矩阵。

### P3（可选）

- [ ] 社区功能
- [ ] 学术引用导出（Markdown/LaTeX/BibTeX）
- [ ] 版本对比工具

---

## 九、总结

**核心价值**：在命理古籍解读这个垂直领域，提供了目前竞品没有的原文+解读+技能的完整内容体系。

**核心问题**：

1. 研究者的核心需求（检索、引用、溯源）没有被满足
2. 技能的差异化价值没有被用户感知

**进化方向**：从"内容平台"到"研究工具"——这不是功能叠加的问题，是产品重心的转移。
