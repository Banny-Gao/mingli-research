## 四、Subagent 执行模式

### 4.1 设计

每批由 1 个独立 subagent 执行，subagent 类型：`general-purpose`。prompt 自包含，SPEC 通过文件引用而非内联。

### 4.2 并行度

**6 并行**（typical max），分轮执行。本书总批数 = 18 批 = 3 轮 × 6。

### 4.3 Subagent Prompt 模板

```text
你是 紫微斗数解读 subagent，负责为本批 N 篇古籍生成 interpretation.md。

【书目】紫微斗数全书
【批号】{batch_id}
【篇目】（源文件路径 books/紫微斗数全书/articles/{篇名}/source.md）
- {篇名1}
- {篇名2}
- ... 共 N 篇

【严格遵循规范】
- 完整解读规范：research-dispute/SPEC-interpretation.md
- 完整通用规范：research-dispute/SPEC-general.md
- 重点：文件禁"升华洞见"；自评分内部不写入；反模板禁止直套
- 紫微特色：术语当场释义，不机械套用八字框架（详见 §七.2）

【输出去向】
books/紫微斗数全书/articles/{篇名}/interpretation.md（每篇 1 个文件）

【完成后报告】
- 已生成文件列表（绝对路径）
- LLM 合规自检评分（内部 0-5，不写入文件）
- 触红线项（如有）
- 任何卡点 / SPEC 改进建议
```

### 4.4 调度流程（每轮）

```
1. 调度方在同一 message 发出 6 个 subagent 调用
2. 每个 subagent 独立完成批内 4-5 篇
3. 全部完成后调度方汇总到 docs/解读日志.md
4. 更新本计划 §三 状态列
5. 下一轮
```

### 4.5 监控指标

- 每批跑批时间（5-10 min 正常）
- 自评合规分均值（≥ 4.5 视为合格）
- 触红线频次（持续高则需重新审视 SPEC）
- 紫微术语误用八字框架的频次（重点监控）

## 六、风险与变更

### 6.1 已知风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 简繁混用（`sourceKeys` 內繁體 `問*`，目录內简体 `问*` 同名异文） | 容易混淆同一篇的两种字形 | LLM 按文件系统目录名（简体）走，繁體属异文需按 SPEC §六.1 处理 |
| 紫微数据规范不同于八字（无"格局"/"用神"等概念） | SPEC §1.2 §1 默认按八字术语设计 | 解读时术语当场释义，紫微特有概念（"星曜""宫位""四化"）不强行套八字术语 |
| `sourceKeys` 漏 12 宫篇目（与 filesystem 不一致） | 漏跑 | 已按 filesystem 89 篇重排，sourceKeys 仅作交叉参考 |
| 模式判据 `mode_of()` 与实际字符数偏差 | 章节层级与预判不符 | LLM 必须以运行时计算为准，不盲信预判 |
| 6 subagent 并行 token 成本 | 单轮消耗约 6× 单批 token | 按需降低并行度（如 3-4），或按轮分批执行 |

### 6.2 计划变更规则

- 篇目调整：在本文件 §三 表中编辑，commit 时附理由
- 批次数量调整：仅当 SPEC 重大修订或篇目整体重排时启用；调整后需重新校验 18 批总和 = 89
- 模式预判偏差：仅注释，不阻断
- 并行度调整：在 §四 记录

### 7.2 紫微特色术语处理

紫微斗数与八字术语体系差异较大，下表为 LLM 解读时的术语对照参考：

| 八字术语 | 紫微对应概念 | 处理建议 |
|----------|-------------|----------|
| 格局 | 格局 / 星格 | 沿用，紫微亦称"格" |
| 用神 | 主星 / 命主 | 不直接对应，紫微用"命宫主星"概念 |
| 月令 | 生月 / 时辰 | 不直接对应，紫微以出生时辰定命宫 |
| 五行 | 五行（仅辅星用） | 紫微主星属性更重要 |
| 旺相休囚 | 入庙 / 落陷 | 紫微特有术语，需当场释义 |

LLM 解读时遇到上表对照关系，**严禁机械套用八字框架**，须按紫微自身理论体系展开。
